索引及MySQL中的使用

Date 2013年09月06日 By masteryoda Category database.
  • MyISAM数据表的数据行是在数据文件里，而索引值是在索引文件里。一个数据表可以有多个索引，所有的索引都存储在同一个索引文件里。索引文件里每一个索引都是由分类的关键记录数组组成的，这些数组用于快速访问数据文件。
  • InnoDB默认使用的是一个表空间，在这个表空间里，它管理着所有InnoDB类型数据表的数据和索引的存储。我们可以通过配置让InnoDB为每个数据表分别创建一个它自己的表空间，但即便如此，一个给定的数据表的数据和索引也是保存在同一个表空间文件里的。

使用索引可以提升单表查询速度，同时在连接多表进行查询时也会发挥作用。

  SELECT t1.i1, t2.i2, t3.i3
  FROM t1 INNER JOIN t2 INNER JOIN t3
  WHERE t1.i1 = t2.i2 AND t2.i2 = t3.i3;

以上查询使用索引的处理步骤如下：

  1. 从数据表t1中依次选择一个数据行
  2. 对数据表t2使用索引，直接找出与数据表t1的值相匹配的数据行，类似地直接找到t3中与t1相匹配的数据行

使用索引的优势

  1. 在查询时把与WHERE子句所给出的条件相匹配的数据行快速找出來
  2. 在连表查询时把与其他数据表里的数据行相匹配的数据行尽快找出來
  3. 对于使用MAX()或MIN()函数的查询，如果数据行带索引，那么其值可以被迅速取出而不用通过遍历数据表查找的方法
  4. 有些索引可以用来迅速完成ORDER BY子句和GROUP BY子句的分类和分组操作

使用索引的缺点

  1. 加快了检索速度缺降低了带索引的数据列里插入/删除以及修改数值的速度
  2. 索引需要占据硬盘空间，多个索引会占据更大空间，这会使你更快的达到数据表的尺寸限制

如何选择索引

  1. 尽量为用来搜索/分类或分组的数据列编制索引，不要为作为输出显示的数据列编制索引

    最适合有索引的数据列是那些在WHERE 子句中出现的数据列/在连接子句中给出的数据列/在ORDER BY或GROUP BY子句中出现的数据列

    根据SELECT关键字仅出现在输出数据列清单里的数据列最好不要有索引

  2. 综合考虑各数据列的维度

    所谓维度是指它所容纳的非重复值的个数。数据列的维度越高，它包含的独一无二的值就越多，重复的值就越少，索引的使用效果就越好。

  3. 对短小的值进行索引

    比较短小的值可以在以下几个方面提升索引的处理性能 - 短小的值可以让比较操作更快的完成，加快索引查询速度 - 短小的值可以让索引的“体积”更小，减少磁盘IO活动 - 短小的键值意味着键缓存里的索引块可以容纳更多的键值，让MySQL可以在内存里同时容纳更多的键。

    InnoDB存储引擎使用的是聚集索引，数据行和主键集中保存在一起，其他索引都是些二级索引——他们保存这主键值和二级索引。查找时先在二级索引里找到一个主键值，再通过它找到相应的数据行。主键值在每一个二级索引里都会出现，所以主键尽量短小要更有好处。

  4. 为字符词值的前缀编索引

  5. 充分利用最左边的前缀

    当创建了一个n个数据列的符合索引时，实际上就创建了MySQL呢个能够使用的n个索引。一个复合索引其实就相当与n个索引，因为索引中最左边的数据列集合可以用于匹配数据行。

    比如有一个复合索引，索引中的数据行是以state/city/zip的顺序存储的。该索引可以用来搜索下面的数据列的组合：

      state, city, zip
  state, city
  state

  6. 适可而止，不要建立过多的索引

  7. 让索引的类型与你打算进行的比较操作的类型保持匹配

    在创建索引的时候，绝大多数存储引擎都会选择他们默认的索引实现 - InnoDB总是使用"B树"索引 - MyISAM也使用“B树”，但在遇到空间数据类型时会改用“R树” - MEMORY存储引擎默认使用HASH索引，但也支持“B树”索引

    对于HASH索引，它在使用“=”或“<=>”操作符进行精确匹配比较操作里的速度很快，但是对于查找一个范围的比较操作里却表现不佳。

    “B树”索引在使用<, <=, =, >=, >, <>, !=和BETWEEN操作符进行的精确比较或范围比较操作里都有很高的效率。如果匹配模式是以一个纯字符串而不是通配符开头的话，“B树”还可以用于使用LIKE操作符进行的模式匹配操作

    可以明确指出创建索引使用的方式，方法是在索引定义里加上USING BTREE

      PRIMARY KEY USING BTREE (id)

    可以让同一个MEMORY数据表同时拥有一个HASH索引和一个B树索引，即使它们都基于同一个数据列也没有问题！
